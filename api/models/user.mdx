## Overview

The User model uses Meteor's built-in Accounts system. Users authenticate with username and password, and are assigned roles that determine their access to API endpoints.

## Schema

### Core Fields

<ResponseField name="_id" type="string">
  MongoDB document ID (auto-generated by Meteor Accounts)
</ResponseField>

<ResponseField name="username" type="string" required>
  Unique username for authentication. Used in:
  - Login endpoint
  - JWT token generation
  - Record assignment (as GESTOR)
</ResponseField>

<ResponseField name="services" type="object">
  Meteor Accounts password service data. Contains hashed password and related authentication metadata.
</ResponseField>

### Profile Object

User profile data is stored in the `profile` object:

<ResponseField name="profile.role" type="string">
  User role. Determines API access permissions.
  
  **Values:**
  - `"management"` - Has access to all management endpoints
  - Other roles may exist but are not documented in the API code
</ResponseField>

<ResponseField name="profile.project" type="string">
  Project assignment (e.g., `"sierra"`). Used to filter records in Meteor methods.
</ResponseField>

## Authentication

### Login Flow

1. Client sends username and password to `/login`
2. Server validates credentials using `Accounts._checkPassword(user, password)`
3. If valid, server generates JWT token: `jwt.sign({ userId: user._id }, JWT_SECRET)`
4. Client receives token for subsequent authenticated requests

### Role-Based Access

The management API endpoints require:
- Valid JWT token in Authorization header
- User must have `profile.role === "management"`

Authentication middleware checks:
```javascript
const user = Meteor.users.findOne(decoded.userId);

if (!user || !user.profile || user.profile.role !== "management") {
  // Access denied
}
```

## User Roles

### Management Role

Users with `profile.role === "management"` can:
- Create and update records via POST `/management`
- Upload photos via POST `/management/photos/:orden`
- Retrieve records via GET `/management/:username`

### Other Roles

The codebase references other potential roles:
- Team leaders (referenced by `leaderId` field in records)
- Admin users (implied by web interface)

These roles are managed through Meteor methods, not the REST API.

## Username in Records

The `username` field is used as the `GESTOR` identifier in records:

```json
{
  "NUMERO_DE_LA_ORDEN": "ORD-12345",
  "GESTOR": "jperez",  // matches user.username
  ...
}
```

To retrieve records for a user:
```bash
GET /management/jperez
```

## Security Notes

<Warning>
  - Passwords are hashed using bcrypt via Meteor Accounts
  - JWT secret is defined in `recordsEndpoints.js` as `"your_jwt_secret_key"` - **CHANGE THIS IN PRODUCTION**
  - JWTs are signed but don't expire by default (unless using refresh token endpoint)
  - All management endpoints verify both token validity and user role
</Warning>
